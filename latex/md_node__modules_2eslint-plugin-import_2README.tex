\hypertarget{md_node__modules_2eslint-plugin-import_2README_autotoc_md11134}{}\doxysection{\texorpdfstring{Resolvers}{Resolvers}}\label{md_node__modules_2eslint-plugin-import_2README_autotoc_md11134}
With the advent of module bundlers and the current state of modules and module syntax specs, it\textquotesingle{}s not always obvious where `import x from \textquotesingle{}module'{\ttfamily should look to find the file behind}module\`{}.

Up through v0.\+10ish, this plugin has directly used substack\textquotesingle{}s \href{https://www.npmjs.com/package/resolve}{\texttt{ {\ttfamily resolve}}} plugin, which implements Node\textquotesingle{}s import behavior. This works pretty well in most cases.

However, webpack allows a number of things in import module source strings that Node does not, such as loaders (`import \textquotesingle{}file!./whatever'{\ttfamily ) and a number of aliasing schemes, such as \mbox{[}}externals\`{}\mbox{]}\+: mapping a module id to a global name at runtime (allowing some modules to be included more traditionally via script tags).

In the interest of supporting both of these, v0.\+11 introduces resolvers.

Currently \href{https://www.npmjs.com/package/eslint-import-resolver-node}{\texttt{ Node}} and \href{https://www.npmjs.com/package/eslint-import-resolver-webpack}{\texttt{ webpack}} resolution have been implemented, but the resolvers are just npm packages, so \href{https://github.com/import-js/eslint-plugin-import/wiki/Resolvers}{\texttt{ third party packages are supported}} (and encouraged!).

You can reference resolvers in several ways (in order of precedence)\+:
