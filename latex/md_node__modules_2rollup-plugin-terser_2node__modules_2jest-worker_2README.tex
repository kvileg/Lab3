\Hypertarget{md_node__modules_2rollup-plugin-terser_2node__modules_2jest-worker_2README_autotoc_md23502}%
 Module for executing heavy tasks under forked processes in parallel, by providing a {\ttfamily Promise} based interface, minimum overhead, and bound workers.

The module works by providing an absolute path of the module to be loaded in all forked processes. Files relative to a node module are also accepted. All methods are exposed on the parent process as promises, so they can be {\ttfamily await}\textquotesingle{}ed. Child (worker) methods can either be synchronous or asynchronous.

The module also implements support for bound workers. Binding a worker means that, based on certain parameters, the same task will always be executed by the same worker. The way bound workers work is by using the returned string of the {\ttfamily compute\+Worker\+Key} method. If the string was used before for a task, the call will be queued to the related worker that processed the task earlier; if not, it will be executed by the first available worker, then sticked to the worker that executed it; so the next time it will be processed by the same worker. If you have no preference on the worker executing the task, but you have defined a {\ttfamily compute\+Worker\+Key} method because you want {\itshape some} of the tasks to be sticked, you can return {\ttfamily null} from it.

The list of exposed methods can be explicitly provided via the {\ttfamily exposed\+Methods} option. If it is not provided, it will be obtained by requiring the child module into the main process, and analyzed via reflection. Node 10 shipped with \href{https://nodejs.org/api/worker_threads.html}{\texttt{ worker-\/threads}}, a "{}threading API"{} that uses Shared\+Array\+Buffers to communicate between the main process and its child threads. This experimental Node feature can significantly improve the communication time between parent and child processes in {\ttfamily jest-\/worker}.

Since {\ttfamily worker\+\_\+threads} are considered experimental in Node, you have to opt-\/in to this behavior by passing {\ttfamily enable\+Worker\+Threads\+: true} when instantiating the worker. Finishes the workers by killing all workers. No further calls can be done to the {\ttfamily Worker} instance.

Returns a Promise that resolves with {\ttfamily \{ force\+Exited\+: boolean \}} once all workers are dead. If {\ttfamily force\+Exited} is {\ttfamily true}, at least one of the workers did not exit gracefully, which likely happened because it executed a leaky task that left handles open. This should be avoided, force exiting workers is a last resort to prevent creating lots of orphans.

{\bfseries{Note\+:}}

{\ttfamily await}ing the {\ttfamily end()} Promise immediately after the workers are no longer needed before proceeding to do other useful things in your program may not be a good idea. If workers have to be force exited, {\ttfamily jest-\/worker} may go through multiple stages of force exiting (e.\+g. SIGTERM, later SIGKILL) and give the worker overall around 1 second time to exit on its own. During this time, your program will wait, even though it may not be necessary that all workers are dead before continuing execution.

Consider deliberately leaving this Promise floating (unhandled resolution). After your program has done the rest of its work and is about to exit, the Node process will wait for the Promise to resolve after all workers are dead as the last event loop task. That way you parallelized computation time of your program and waiting time and you didn\textquotesingle{}t delay the outputs of your program unnecessarily.\hypertarget{md_node__modules_2rollup-plugin-terser_2node__modules_2jest-worker_2README_autotoc_md23524}{}\doxysubsection{\texorpdfstring{Worker IDs}{Worker IDs}}\label{md_node__modules_2rollup-plugin-terser_2node__modules_2jest-worker_2README_autotoc_md23524}
