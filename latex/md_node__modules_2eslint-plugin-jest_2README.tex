This is included in all configs shared by this plugin, so can be omitted if extending them.\hypertarget{md_node__modules_2eslint-plugin-jest_2README_autotoc_md11507}{}\doxysubsubsection{\texorpdfstring{Jest $<$tt$>$version$<$/tt$>$ setting}{Jest <tt>version</tt> setting}}\label{md_node__modules_2eslint-plugin-jest_2README_autotoc_md11507}
The behaviour of some rules (specifically \doxysectlink{md_node__modules_2eslint-plugin-jest_2docs_2rules_2no-deprecated-functions}{\`{}no-\/deprecated-\/functions\`{}}{0}) change depending on the version of Jest being used.

By default, this plugin will attempt to determine to locate Jest using {\ttfamily require.\+resolve}, meaning it will start looking in the closest {\ttfamily node\+\_\+modules} folder to the file being linted and work its way up.

Since we cache the automatically determined version, if you\textquotesingle{}re linting sub-\/folders that have different versions of Jest, you may find that the wrong version of Jest is considered when linting. You can work around this by providing the Jest version explicitly in nested ESLint configs\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\{}
\DoxyCodeLine{\ \ "{}settings"{}:\ \{}
\DoxyCodeLine{\ \ \ \ "{}jest"{}:\ \{}
\DoxyCodeLine{\ \ \ \ \ \ "{}version"{}:\ 27}
\DoxyCodeLine{\ \ \ \ \}}
\DoxyCodeLine{\ \ \}}
\DoxyCodeLine{\}}

\end{DoxyCode}

\begin{DoxyCode}{0}
\DoxyCodeLine{module.exports\ =\ \{}
\DoxyCodeLine{\ \ settings:\ \{}

This plugin also exports a configuration named {\ttfamily style}, which adds some stylistic rules, such as {\ttfamily prefer-\/to-\/be-\/null}, which enforces usage of {\ttfamily to\+Be\+Null} over {\ttfamily to\+Be(null)}.

To enable this configuration use the {\ttfamily extends} property in your {\ttfamily .eslintrc} config file\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\{}
\DoxyCodeLine{\ \ "{}extends"{}:\ ["{}plugin:jest/style"{}]}
\DoxyCodeLine{\}}

\end{DoxyCode}

\hypertarget{md_node__modules_2eslint-plugin-jest_2README_autotoc_md11513}{}\doxysubsection{\texorpdfstring{Type\+Script Rules}{TypeScript Rules}}\label{md_node__modules_2eslint-plugin-jest_2README_autotoc_md11513}
In addition to the above rules, this plugin also includes a few advanced rules that are powered by type-\/checking information provided by Type\+Script.

In order to use these rules, you must be using {\ttfamily @typescript-\/eslint/parser} \& adjust your eslint config as outlined \href{https://github.com/typescript-eslint/typescript-eslint/blob/master/docs/getting-started/linting/TYPED_LINTING.md}{\texttt{ here}}

Note that unlike the type-\/checking rules in {\ttfamily @typescript-\/eslint/eslint-\/plugin}, the rules here will fallback to doing nothing if type information is not available, meaning its safe to include them in shared configs that could be used on Java\+Script and Type\+Script projects.

Also note that {\ttfamily unbound-\/method} depends on {\ttfamily @typescript-\/eslint/eslint-\/plugin}, as it extends the original {\ttfamily unbound-\/method} rule from that plugin.
