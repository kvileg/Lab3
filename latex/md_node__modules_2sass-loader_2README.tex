{\ttfamily sass-\/loader} requires you to install either \href{https://github.com/sass/dart-sass}{\texttt{ Dart Sass}}, \href{https://github.com/sass/node-sass}{\texttt{ Node Sass}} on your own (more documentation can be found below) or \href{https://github.com/sass/embedded-host-node}{\texttt{ Sass Embedded}}.

This allows you to control the versions of all your dependencies, and to choose which Sass implementation to use.

\begin{quote}
ℹ️ We highly recommend using \href{https://github.com/sass/dart-sass}{\texttt{ Dart Sass}}. \end{quote}
\begin{quote}
⚠ \href{https://github.com/sass/node-sass}{\texttt{ Node Sass}} does not work with \href{https://classic.yarnpkg.com/en/docs/pnp/}{\texttt{ Yarn PnP}} feature and doesn\textquotesingle{}t support \href{https://sass-lang.com/documentation/at-rules/use}{\texttt{ @use rule}}. \end{quote}
\begin{quote}
⚠ \href{https://github.com/sass/embedded-host-node}{\texttt{ Sass Embedded}} is experimental and in {\ttfamily beta}, therefore some features may not work \end{quote}
Chain the {\ttfamily sass-\/loader} with the \href{https://github.com/webpack-contrib/css-loader}{\texttt{ css-\/loader}} and the \href{https://github.com/webpack-contrib/style-loader}{\texttt{ style-\/loader}} to immediately apply all styles to the DOM or the \href{https://github.com/webpack-contrib/mini-css-extract-plugin}{\texttt{ mini-\/css-\/extract-\/plugin}} to extract it into a separate file.

Finally run {\ttfamily webpack} via your preferred method.\hypertarget{md_node__modules_2sass-loader_2README_autotoc_md25288}{}\doxysubsubsection{\texorpdfstring{Resolving $<$tt$>$import$<$/tt$>$ at-\/rules}{Resolving <tt>import</tt> at-rules}}\label{md_node__modules_2sass-loader_2README_autotoc_md25288}
Webpack provides an \href{https://webpack.js.org/concepts/module-resolution/}{\texttt{ advanced mechanism to resolve files}}.

The {\ttfamily sass-\/loader} uses Sass\textquotesingle{}s custom importer feature to pass all queries to the Webpack resolving engine. Thus you can import your Sass modules from {\ttfamily node\+\_\+modules}.


\begin{DoxyCode}{0}
\DoxyCodeLine{@import\ "{}bootstrap"{};}

\end{DoxyCode}


Using {\ttfamily \texorpdfstring{$\sim$}{\string~}} is deprecated and can be removed from your code ({\bfseries{we recommend it}}), but we still support it for historical reasons. Why can you remove it? The loader will first try to resolve {\ttfamily @import} as a relative path. If it cannot be resolved, then the loader will try to resolve {\ttfamily @import} inside \href{https://webpack.js.org/configuration/resolve/\#resolvemodules}{\texttt{ {\ttfamily node\+\_\+modules}}}.

Prepending module paths with a {\ttfamily \texorpdfstring{$\sim$}{\string~}} tells webpack to search through \href{https://webpack.js.org/configuration/resolve/\#resolvemodules}{\texttt{ {\ttfamily node\+\_\+modules}}}.


\begin{DoxyCode}{0}
\DoxyCodeLine{@import\ "{}\string~bootstrap"{};}

\end{DoxyCode}


It\textquotesingle{}s important to prepend it with only {\ttfamily \texorpdfstring{$\sim$}{\string~}}, because {\ttfamily \texorpdfstring{$\sim$}{\string~}/} resolves to the home directory. Webpack needs to distinguish between {\ttfamily bootstrap} and {\ttfamily \texorpdfstring{$\sim$}{\string~}bootstrap} because CSS and Sass files have no special syntax for importing relative files. Writing {\ttfamily @import "{}style.\+scss"{}} is the same as {\ttfamily @import "{}./style.\+scss"{};}\hypertarget{md_node__modules_2sass-loader_2README_autotoc_md25289}{}\doxysubsubsection{\texorpdfstring{Problems with $<$tt$>$url(...)$<$/tt$>$}{Problems with <tt>url(...)</tt>}}\label{md_node__modules_2sass-loader_2README_autotoc_md25289}
Since Sass implementations don\textquotesingle{}t provide \href{https://github.com/sass/libsass/issues/532}{\texttt{ url rewriting}}, all linked assets must be relative to the output.


\begin{DoxyItemize}
\item If you pass the generated CSS on to the {\ttfamily css-\/loader}, all urls must be relative to the entry-\/file (e.\+g. {\ttfamily main.\+scss}).
\item If you\textquotesingle{}re just generating CSS without passing it to the {\ttfamily css-\/loader}, it must be relative to your web root.
\end{DoxyItemize}

You will be disrupted by this first issue. It is natural to expect relative references to be resolved against the {\ttfamily .sass}/{\ttfamily .scss} file in which they are specified (like in regular {\ttfamily .css} files).

Thankfully there are a two solutions to this problem\+:


\begin{DoxyItemize}
\item Add the missing url rewriting using the \href{https://github.com/bholloway/resolve-url-loader}{\texttt{ resolve-\/url-\/loader}}. Place it before {\ttfamily sass-\/loader} in the loader chain.
\item Library authors usually provide a variable to modify the asset path. \href{https://github.com/twbs/bootstrap-sass}{\texttt{ bootstrap-\/sass}} for example has an {\ttfamily \$icon-\/font-\/path}.
\end{DoxyItemize}\hypertarget{md_node__modules_2sass-loader_2README_autotoc_md25290}{}\doxysubsection{\texorpdfstring{Options}{Options}}\label{md_node__modules_2sass-loader_2README_autotoc_md25290}

The special {\ttfamily implementation} option determines which implementation of Sass to use.

By default the loader resolve the implementation based on your dependencies. Just add required implementation to {\ttfamily package.\+json} ({\ttfamily sass} or {\ttfamily node-\/sass} package) and install dependencies.

Note that when using {\ttfamily sass} ({\ttfamily Dart Sass}), {\bfseries{synchronous compilation is twice as fast as asynchronous compilation}} by default, due to the overhead of asynchronous callbacks. To avoid this overhead, you can use the \href{https://www.npmjs.com/package/fibers}{\texttt{ fibers}} package to call asynchronous importers from the synchronous code path.

We automatically inject the \href{https://github.com/laverdet/node-fibers}{\texttt{ {\ttfamily fibers}}} package (setup {\ttfamily sass\+Options.\+fiber}) for {\ttfamily Node.\+js} less v16.\+0.\+0 if is possible (i.\+e. you need install the \href{https://github.com/laverdet/node-fibers}{\texttt{ {\ttfamily fibers}}} package).

\begin{quote}
Fibers is not compatible with {\ttfamily Node.\+js} v16.\+0.\+0 or later (\href{https://github.com/laverdet/node-fibers}{\texttt{ see introduction to readme}}). \end{quote}

ℹ️ Access to the \href{https://webpack.js.org/api/loaders/\#the-loader-context}{\texttt{ loader context}} inside the custom importer can be done using the {\ttfamily this.\+webpack\+Loader\+Context} property. \end{quote}
